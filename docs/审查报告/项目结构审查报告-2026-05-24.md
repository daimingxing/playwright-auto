# 项目结构审查报告（2026-05-24）

## 审查范围

本报告只评估目录结构、模块边界、文档和维护成本，不重复展开具体 bug。事实来源包括当前文件系统、`README.md`、`docs/agent-code-map.md`、`docs/agent-commands.md`、`package.json`、后端 `server/src`、前端 `web/src`、共享模块 `shared` 和测试目录。

## 总体结论

当前结构不需要大规模重构。项目按 `web`、`server`、`shared`、`tests`、`docs` 分层，业务入口清晰；后端路由、服务和基础库分离基本合理；前端页面、API、状态工具和纯函数工具也有边界。

值得优化的是“结构一致性”和“可维护性”：CodeGraph 索引滞后、开发脚本文档与真实文件系统不一致、部分前端页面单文件过大、运行/实测/录制三条 Playwright 子进程链路有重复模式、配置类型在 `shared` 中没有完整表达。这些问题尚未到必须重构的程度，但会影响后续迭代速度和审查可信度。

## 当前结构优点

- `shared/types.ts`、`shared/url.ts`、`shared/case-review.ts` 承担前后端共享契约，减少重复规则。
- `server/src/routes` 主要处理 HTTP 入参和响应，核心逻辑下沉到 `server/src/services` 与 `server/src/lib`。
- `server/src/lib/guard.ts`、`path.ts`、`http-error.ts` 已形成路径和错误语义边界，利于继续加固。
- `web/src/api` 封装请求，`web/src/pages/*.ts` 提取部分页面纯逻辑，便于单元测试。
- 测试目录按 `server`、`web`、`shared`、`smoke` 分组，覆盖了大量核心逻辑。
- `docs/agent-code-map.md` 与 `docs/agent-commands.md` 对 AI 协作很有帮助，能降低定位成本。

## 结构隐患与优化建议

### S1：CodeGraph 索引与真实工作树不一致

- 现象：CodeGraph 显示 `scripts/dev.mjs`、`scripts/wait-for-server.mjs` 和多个 `git_stats_*.cjs`，但 `rg --files` 和直接读取确认当前工作树没有这些脚本文件。
- 影响：结构问题如果只信 CodeGraph，容易把不存在的脚本当作真实入口。
- 建议：重新初始化或刷新 CodeGraph 索引；在结构性审查中保留“CodeGraph 仅作结构入口，关键结论以文件系统确认”的规则。

### S1：启动脚本结构退化，`scripts/` 目录只剩安装检查

- 现状：`scripts/` 当前只有 `install-browsers.mjs`；`package.json` 的 `dev` 直接并行启动前后端。
- 影响：启动编排散落在 `package.json` 字符串里，无法承载健康检查、端口检测、Windows 友好错误等逻辑。
- 建议：恢复一个专门的开发启动脚本，例如 `scripts/dev.mjs` 或 `scripts/wait-for-server.mjs`，让 `package.json` 只保留短命令。脚本应读取 `playwright-auto.config.json` 的端口，避免硬编码 `3001`。

### S2：前端大页面仍承担过多职责

- 观察：`CaseEditor.vue`、`RunCenter.vue`、`ProjectList.vue` 同时包含数据加载、状态同步、业务判断、弹窗流程和大量样式。
- 影响：继续扩展时会出现改 UI 影响业务流程、测试只能覆盖拆出的少量纯函数的问题。
- 建议：保持现有页面入口不变，渐进拆分 composable 或页面服务：
  - `useRunAuth`：运行中心登录态、会话保存、环境切换。
  - `useRunReports`：报告列表、刷新、打开、导出、删除。
  - `useProjectEnvs`：项目列表页环境增删改。
  - 复杂表格样式继续留在页面，避免过度抽象。

### S2：Playwright 子进程链路有重复模式

- 位置：`runner.ts`、`practical-review.ts`、`record-session.ts`
- 现状：三处都要组装环境变量、收集输出、处理退出码、决定是否使用登录态和离线浏览器。
- 影响：后续改 Playwright CLI 策略、输出摘要、进程超时或清理逻辑时，需要多处同步。
- 建议：在 `server/src/services/playwright-cli.ts` 旁边扩展小型 helper，例如 `runPlaywrightTask`，统一 stdout/stderr 收集、退出码处理和环境变量合并。录制的交互式 codegen 可保留单独函数。

### S2：配置类型没有完整共享

- 现状：`shared/types.ts` 里的 `AppConfig` 只包含 `steps`，而真实配置还包含 `server`、`web`、`runner`。
- 影响：`server/src/lib/app-config.ts`、`web/vite.config.ts`、前端 API 类型各自描述一部分配置，字段变更时不容易被类型系统完整提醒。
- 建议：把完整运行配置拆成共享类型，例如 `ServerConfig`、`WebConfig`、`RunnerConfig`、`StepConfig`，再由后端返回前端需要的安全子集。注意不要把服务端私有字段直接暴露到 `/api/app-config`。

### S3：文档索引有价值，但需要补“审查报告入口”

- 现状：`README.md` 和 `docs/agent-code-map.md` 已记录核心代码入口，但没有列出 `docs/审查报告/` 作为结构性审查结果入口。
- 影响：后续审查报告容易被遗忘，结论沉淀后不易再次使用。
- 建议：在下一次文档维护时，把 `docs/审查报告/` 加入 README 或 `docs/agent-code-map.md` 的文档索引。

### S3：测试结构健康，但端到端覆盖较轻

- 现状：单元测试和接口测试覆盖良好，`tests/smoke/startup.test.ts` 只验证 `/health`。
- 影响：前后端能力不一致的问题，例如“后端可无登录态运行但前端阻断”，更容易漏掉。
- 建议：增加少量前端流程层测试，不必完整启动浏览器。优先覆盖运行中心按钮启用条件、无登录态提示但允许运行、报告异常 URL 错误展示。

## 不建议现在做的大改

- 不建议把 `server/src/lib` 与 `server/src/services` 大规模重排。当前边界虽然不完美，但已有文档和测试围绕这些路径建立。
- 不建议把所有前端页面一次性拆成很多小组件。更合适的是先抽纯逻辑和 composable，等功能稳定后再拆视觉组件。
- 不建议引入数据库或复杂状态管理。当前本地文件系统模型符合工具定位，优先补原子写入和错误提示即可。
- 不建议为审查单独生成大量新目录。继续使用 `docs/审查报告/` 承载结果即可。

## 建议路线

1. 先修复 bug 报告中 P1/P2 项，减少真实用户阻断。
2. 刷新 CodeGraph，并把脚本真实入口和文档入口对齐。
3. 小步抽出运行中心 composable，优先拆登录态和报告列表。
4. 抽 Playwright 子进程公共 helper，统一超时、输出和清理策略。
5. 完善共享配置类型，但保持 `/api/app-config` 只返回前端需要字段。
