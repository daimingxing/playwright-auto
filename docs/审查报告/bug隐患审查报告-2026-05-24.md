# Bug 与隐患审查报告（2026-05-24）

## 审查范围

本次审查覆盖本地配置、后端 API、文件系统边界、Playwright 子进程、录制和运行链路、前端 API 与运行中心交互、测试和构建脚本。事实来源以当前工作树为准，CodeGraph 仅作结构入口参考；发现其索引存在滞后，已记录到 `docs/problem-record.md`。

## 总体结论

项目当前自动检查整体健康，现有 225 条测试全部通过，生产构建也能完成。核心安全边界已有基础防护：路径参数集中校验，CORS 默认限制本机前端来源，Playwright 运行与实测检查使用 `process.execPath` 启动本地 CLI 且 `shell:false`。

仍建议优先处理 3 个真实隐患：报告静态资源畸形 URL 会返回 500、运行中心前端强制登录态导致无登录项目无法运行、开发启动脚本未等待后端健康检查。另有若干中低优先级改进项，主要集中在配置错误可观测性、会话生命周期和写文件可靠性。

## 已确认问题

### P1：报告静态资源畸形编码会触发 500

- 位置：`server/src/routes/runs.ts` 的 `resolveReportFile`
- 现象：访问报告静态资源时，如果 URL 包含畸形百分号编码，`decodeURIComponent` 会抛出 `URIError`，随后进入兜底 500。
- 复现证据：一次性脚本请求 `/api/projects/crm/runs/<runId>/report/%E0%A4%A`，实际返回 `{"status":500,"body":{"message":"服务内部错误"}}`。
- 影响：浏览器、代理或手动复制出的坏链接会把客户端输入错误表现成服务内部错误，误导排查，也让错误通道不一致。
- 建议：在 `resolveReportFile` 中捕获解码错误并返回 `400 报告路径不合法`；补充 API 测试覆盖畸形编码和路径穿越两类边界。

### P1：运行中心前端强制登录态，和后端能力不一致

- 位置：`web/src/pages/run-center.ts`、`web/src/pages/RunCenter.vue`、`server/src/services/runner.ts`
- 现象：前端 `canStartRun` 要求 `hasAuth` 为真，`startRun` 也会在无登录态时直接提示“请先保存项目登录态”；但后端 `runProject` 实际在无登录态时传空 `PLAYWRIGHT_STORAGE_STATE`，支持公开站点或无需登录项目运行。
- 影响：不需要登录的项目无法从 UI 启动测试，只能绕过前端调用 API，属于功能阻断。
- 建议：把登录态从“运行前置条件”改为“可选增强”。运行按钮只要求已选用例且未运行中；未保存登录态时给出提示但允许继续。同步调整 `tests/web/run-center.test.ts`。

### P2：开发启动仍是前后端并行，未等待后端健康检查

- 位置：`package.json` 的 `dev`，`web/vite.config.ts` 的 `/api` 与 `/health` 代理
- 现状：`npm run dev` 为 `concurrently -k "npm:dev:server" "npm:dev:web"`，当前文件系统没有 `scripts/wait-for-server.mjs` 或等价启动闸门。
- 影响：前端先启动并请求 `/api` 时，Vite proxy 可能遇到后端未监听导致 `ECONNREFUSED`，开发体验不稳定。记忆中曾处理过这类问题，但当前工作树未保留该脚本接入。
- 建议：恢复或新增健康检查等待脚本，并把 `dev:web` 改为等待 `http://localhost:3001/health` 成功后再启动 Vite；文档同步说明该启动顺序。

### P2：配置文件解析失败会静默回退默认值

- 位置：`server/src/lib/app-config.ts` 的 `readFileConfig`
- 现象：配置文件不存在时回退默认值是合理的，但 JSON 语法错误、字段类型错误也会被静默吞掉，运行时看起来像“配置未生效”。
- 影响：端口、数据目录、CORS 来源、并发上限等关键配置错误难以定位，容易出现事实来源不清。
- 建议：区分“文件不存在”和“文件存在但解析失败”。解析失败至少输出明确错误并终止启动，或返回可见诊断。字段越界可继续回退，但建议记录告警。

### P2：登录和录制会话没有过期清理

- 位置：`server/src/services/auth-session.ts`、`server/src/services/record-session.ts`
- 现象：两个模块都用进程内 `Map` 保存会话。用户打开浏览器后如果不保存，或录制启动后不停止，会话会一直留在内存中；录制还可能保留子进程和临时目录。
- 影响：长期使用本地服务时可能累积无效会话、浏览器进程或临时文件，尤其适合桌面工具场景反复操作时触发。
- 建议：为会话增加 TTL 和清理函数；启动新会话时清理同项目同用例旧会话；停止失败也要删除 `Map` 并清理临时目录。

### P3：JSON 写入不是原子写，异常中断可能留下半文件

- 位置：`server/src/lib/fs.ts` 的 `writeJson`
- 现象：当前直接 `writeFile` 覆盖 `case.json`、`project.json`、`run.json` 等核心数据。
- 影响：进程中断或磁盘异常可能留下截断 JSON，之后 `readJson` 抛错并被部分接口表现为 500。
- 建议：改为写临时文件后 `rename` 覆盖；读取 JSON 失败时按数据损坏返回更明确错误。该项优先级低于前述功能阻断，但对本地持久化可靠性有价值。

## 已验证为相对稳妥的点

- CORS 默认拒绝非配置来源，且允许来源匹配前会归一化尾随斜杠。
- 路径参数通过 `guard.ts` 在进入文件系统路径前校验，项目、用例、运行、实测检查目录都有基础边界。
- 项目导出排除了 `auth/**` 和 `*.storageState.json`，不会直接带出登录态文件。
- Playwright 运行、实测检查、录制和 Windows 结束进程树均使用 `shell:false`。
- `RunError` 会在通用 500 之前返回 400，并带 `reportPath`、`reportUrl`。

## 验证结果

- `npm run typecheck`：通过。
- `npm run test`：通过，25 个测试文件、225 条测试全部通过。
- `npm run build`：通过；存在 Rollup 对 `@vueuse/core` 注释的非阻断警告。
- 一次性边界复现：报告畸形资源 URL 返回 500，确认问题存在。

## 建议处理顺序

1. 修复报告 URL 解码错误为 400，并补测试。
2. 放开运行中心无登录态运行，保留提示但不阻断。
3. 恢复开发启动健康检查等待，减少本地启动竞态。
4. 改造配置解析错误提示和会话 TTL。
5. 评估 JSON 原子写入，适合与存储层测试一起做。
