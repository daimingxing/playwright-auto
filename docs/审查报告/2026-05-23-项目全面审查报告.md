# 项目全面审查报告

审查日期：2026-05-23

审查范围：`data/` 目录是保存的项目用例和运行产物，本次按要求不审查；其余项目源码、测试、配置、文档入口均纳入审查。

## 总体结论

当前项目整体框架方向是合理的：Vue 3 前端、本地 Node/Express API、共享 TypeScript 类型、文件系统持久化、Vitest 测试基线，符合本地 Playwright 自动化平台首版形态。代码分层也基本清楚：前端页面通过 `web/src/api/*` 调用接口；后端路由在 `server/src/routes/*`，业务逻辑在 `server/src/services/*` 和 `server/src/lib/*`；共享协议在 `shared/types.ts`。

不建议现在进行大规模重构。当前更需要先修复 2 个已确认 bug，再补齐路径参数校验、生成代码转义、前端构建拆包和状态管理边界。等运行、录制、实测检查继续扩展后，再考虑把存储层和 API 校验抽成更统一的模块。

## 已确认问题

### P1：删除到回收站遇到同名目录会在 Windows 下失败

位置：

- `server/src/lib/case-store.ts:86`
- `server/src/lib/case-store.ts:93`
- `tests/server/fs-store.test.ts:59`
- `tests/server/fs-store.test.ts:75`
- `tests/server/api-cases.test.ts:48`

现象：

`deleteCase` 当前直接执行：

```ts
await movePath(getCasePath(projectKey, caseKey), getTrashPath(projectKey, caseKey));
```

如果 `trash/<caseKey>` 已经存在，再删除同名 `cases/<caseKey>`，Windows 上会抛出 `EPERM: operation not permitted, rename ...`。本次用系统临时目录复现成功：

```json
{"ok":false,"name":"Error","message":"EPERM: operation not permitted, rename '...\\cases\\case-1' -> '...\\trash\\case-1'"}
```

影响：

历史旧编号用例、恢复后再次删除、或手工导入同名目录时，删除会失败。现有测试只覆盖首次删除进入回收站，没有覆盖目标回收站目录已存在的场景。

建议：

恢复唯一回收站目标名策略，例如 `case-1`、`case-1-1`、`case-1-2`，并新增服务层和 API 层回归测试。恢复时继续按 `case.json` 中的原始 key 或唯一 key 处理，避免目录名和用例 key 混乱。

### P1：用例生成器遇到单引号会生成非法 TypeScript

位置：

- `server/src/services/case-generator.ts:149`
- `tests/server/case-generator.test.ts:11`
- `tests/server/case-generator.test.ts:61`

现象：

`case-generator.ts` 中的 `quote` 先 `JSON.stringify`，再把外层双引号替换成单引号：

```ts
return JSON.stringify(value).replace(/^"|"$/g, "'");
```

当用例名称、输入值或 locator 参数包含单引号时，会生成非法代码。本次最小复现输出：

```ts
test('O'Reilly 登录', async ({ page }) => {
  await page.getByLabel('User's name').fill('O'Reilly');
});
```

影响：

测试人员录制或手工编辑包含英文缩写、英文姓名、带单引号文本的业务步骤后，生成的 `case.spec.ts` 可能无法编译或运行。实测检查生成器已使用原始 `JSON.stringify`，没有同类问题。

建议：

普通用例生成器也改为使用合法 JS 字符串字面量输出，不要替换外层引号。测试断言可以调整为包含双引号输出，新增覆盖用例名、输入值、locator 中单引号的测试。

## 结构性隐患

### P2：路由参数没有统一校验，路径边界依赖分散

位置：

- `server/src/lib/path.ts:21`
- `server/src/lib/path.ts:28`
- `server/src/lib/path.ts:35`
- `server/src/lib/path.ts:42`
- `server/src/routes/projects.ts:31`
- `server/src/routes/cases.ts:98`
- `server/src/routes/runs.ts:57`
- `server/src/lib/schema.ts:3`

说明：

项目创建时有 `projectKeySchema`，运行编号有 `assertRunId`，实测检查编号有 `assertReviewId`，但大多数路由参数进入 `getProjectPath`、`getCasePath`、`getTrashPath` 前没有统一校验。`path.ts` 直接使用 `resolve(root, userParam)`，安全边界主要靠 Express 路由和部分调用点自己限制。

本次验证没有确认出任意目录删除；例如 `/api/projects/crm/trash/%2e%2e` 的 DELETE 返回 404。但 GET 类畸形路径出现过被路由规范化到父级接口并返回项目配置的现象，说明边界不够清晰。

建议：

增加统一的参数断言函数，例如 `assertProjectKey`、`assertCaseKey`、`assertEnvKey`、`assertRunId`，并在路由入口或路径函数内执行。补充 `%2e%2e`、斜杠编码、空值、超长值等畸形 URL 测试。

### P2：子进程使用 `shell: true`，后续扩展时容易引入命令注入风险

位置：

- `server/src/services/runner.ts:71`
- `server/src/services/runner.ts:82`
- `server/src/services/practical-review.ts:186`
- `server/src/services/practical-review.ts:188`
- `server/src/services/record-session.ts:72`
- `server/src/services/record-session.ts:78`

说明：

录制模块已经使用 `process.execPath` + Playwright CLI 路径 + `shell: false`，相对稳妥。运行和实测检查仍通过 `spawn('npx', ..., { shell: true })`。当前参数主要来自受控 key 和内部路径，风险可控，但后续如果把更多用户输入加入参数，会放大风险。

建议：

把运行和实测检查也改为不经 shell 的方式：优先解析本地 Playwright CLI，使用 `process.execPath` 执行 CLI 文件；或至少把所有可变参数维持在数组参数中，并禁止拼接命令字符串。新增测试断言 `shell: false`。

### P2：本地 API CORS 放开所有来源

位置：

- `server/src/app.ts:17`

说明：

服务端使用 `cors({ origin: true })`，对本地工具而言开发期方便，但也意味着任意网页可以从浏览器发起本地 API 请求。该 API 能创建项目、保存登录态、运行测试、删除报告和用例。虽然这是本地工具，不等同公网服务，但本地浏览器访问恶意网页时存在跨站调用本地服务的风险。

建议：

默认只允许 `http://localhost:5173`、`http://127.0.0.1:5173` 和配置中的前端地址。若需要开放，放到配置项里显式启用。

### P2：全局错误处理都返回 400，排查和前端语义不够准确

位置：

- `server/src/app.ts:35`
- `server/src/app.ts:43`

说明：

现在所有非 `RunError` 错误都转成 400。文件不存在、报告不存在、路由参数非法、内部读写错误、JSON 解析错误都混在同一个状态码里。对测试人员页面提示影响不大，但对接口调试、自动化判断和未来错误追踪不利。

建议：

引入轻量 `HttpError` 或错误映射：不存在返回 404，参数校验返回 400，冲突返回 409，内部错误返回 500。Zod 校验错误可统一转成用户友好信息。

### P3：前端 API 固定请求 `localhost:3001`

位置：

- `web/src/api/http.ts:51`
- `web/src/api/http.ts:57`
- `web/vite.config.ts:10`
- `web/vite.config.ts:12`

说明：

前端开发代理已经把 `/api` 转发到后端，但 `resolveApiUrl` 又把 `/api` 固定改成 `http://localhost:3001`。这在本地默认端口没问题，但会削弱 Vite 代理、反向代理、改端口部署或使用 `127.0.0.1` 访问时的灵活性。

建议：

开发模式优先使用相对路径 `/api`，需要跨端口时通过环境变量配置 API base。下载和报告打开也保持同一套 URL 解析逻辑。

### P3：前端引入 Pinia 但没有使用，跨页面状态仍以局部状态为主

位置：

- `package.json:24`
- `web/src/main.ts:7`
- `web/src/pages/RunCenter.vue:31`
- `web/src/pages/RunCenter.vue:44`
- `web/src/pages/ProjectList.vue:16`
- `web/src/pages/CaseEditor.vue:70`

说明：

当前页面规模下，局部 `ref/reactive` 可以工作。但项目已引入 Pinia 却没有 `createPinia` 或 `defineStore`，运行环境、用例选择、登录态状态、项目列表等状态分散在页面内部。随着功能增加，容易出现页面切换后状态丢失、重复请求和状态不同步。

建议：

暂不为此做大重构。等运行中心、项目环境、用例编辑继续扩展时，可以先抽一个 `projectStore` 或 `runStore`，只承接跨页面状态，不把所有表单状态都搬进去。

### P3：前端构建单包过大

位置：

- `web/src/main.ts:2`
- `web/vite.config.ts:20`

说明：

`npm run build` 通过，但 Vite 报告主 JS 包约 `1,072.21 kB`，gzip 后约 `351.18 kB`，超过默认 500 kB 警告。主要原因是全量引入 Element Plus 和页面都打进一个入口包。

建议：

使用路由级动态导入拆分页面，或配置 `manualChunks` 将 Vue、Element Plus、Playwright 相关前端模块分块。Element Plus 也可以考虑按需引入。

## 架构评价

### 合理的部分

- 前后端边界清楚，Vue 页面没有直接访问文件系统。
- 共享类型集中在 `shared/types.ts`，减少前后端协议漂移。
- 后端路由、业务服务、文件存储大体分层合理。
- 测试覆盖面较好，当前有服务层、API 层、前端纯函数和启动 smoke 测试。
- 本地 Web + Node 的首版形态适合测试人员使用，也便于迁移生成的 Playwright 用例。

### 不建议立刻大改的部分

- 不建议把文件存储立刻换成数据库。当前项目规模和本地工具定位下，文件系统模型仍然合适。
- 不建议把所有前端状态一次性迁到 Pinia。先处理真正跨页面的项目/运行状态即可。
- 不建议把路由层、服务层、存储层做大范围重命名或拆包。现有目录对后续 Agent 定位已经比较友好。

### 建议优先做的小重构

1. 建立统一参数校验和路径边界模块，把 projectKey、caseKey、envKey、runId、reviewId 的校验集中起来。
2. 建立统一代码生成字符串工具，普通用例生成器和实测检查生成器共用，避免转义策略分裂。
3. 把子进程启动封装成 `runPlaywrightCli` 一类内部函数，统一 cwd、env、shell 策略、输出收集和错误格式。
4. 抽一个轻量 HTTP 错误类型，统一 API 错误状态码与响应体。
5. 运行中心状态后续复杂化时，再引入 Pinia store。
