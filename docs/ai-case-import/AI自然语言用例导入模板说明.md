# AI 自然语言用例导入模板说明

## 使用场景

本模板用于批量整理自然语言测试用例。测试人员按固定工作表填写用例、步骤和测试数据后，将 Excel 文件上传到平台，由平台生成可编辑的草稿用例。

每条用例必须有目标页面 URL。平台会从该地址进入页面，因此不要只写菜单名称、模块名称或业务入口说明。

## 工作表说明

模板包含四个工作表：

| 工作表 | 用途 |
|---|---|
| 用例清单 | 一行填写一条用例的基本信息。 |
| 步骤明细 | 一行填写一个操作步骤，按步骤序号排列。 |
| 测试数据 | 一行填写一个可被步骤引用的数据项。 |
| 填写说明 | 在 Excel 内提供简要填写提示。 |

## 用例清单填写规则

| 字段 | 填写规则 |
|---|---|
| 用例编号 | 必填，同一文件内唯一，例如 TC001。 |
| 用例名称 | 必填，使用简短业务名称，例如“新增用户”。 |
| 目标页面URL | 必填，填写用例开始操作的页面地址。 |
| 前置条件 | 可选，填写登录状态、账号权限、已有数据等条件。 |
| 预期结果 | 必填，只写业务预期，不写技术术语。 |
| 备注 | 可选，填写补充说明。 |

预期结果建议写成自然语言，例如“页面提示保存成功，用户列表中显示新建用户”。不要写成平台内部处理方式。

## 步骤明细填写规则

操作过程写在步骤明细，不要写成一整段。每个动作拆成独立步骤，步骤序号从 1 开始递增。

| 字段 | 填写规则 |
|---|---|
| 用例编号 | 必填，必须对应用例清单中的用例编号。 |
| 步骤序号 | 必填，同一用例内按 1、2、3 递增。 |
| 操作描述 | 必填，描述当前步骤要做什么。 |
| 目标对象 | 可选，填写页面上的对象名称，例如“新增按钮”“用户名称输入框”。 |
| 数据引用 | 可选，如步骤需要数据，填写测试数据中的数据标识。 |
| 备注 | 可选，填写补充说明。 |

如果点击后打开弹窗、跳转页面、打开新窗口，要写在操作描述中，例如“点击新增按钮，打开新增用户弹窗”。

## 测试数据填写规则

测试数据用于让步骤引用具体输入值或查询条件。数据引用必须和测试数据中的数据标识一致。

| 字段 | 填写规则 |
|---|---|
| 用例编号 | 必填，必须对应用例清单中的用例编号。 |
| 数据标识 | 必填，同一用例内唯一，例如 username。 |
| 数据名称 | 必填，填写给测试人员看的业务名称。 |
| 数据值 | 必填，填写实际使用的数据。 |
| 说明 | 可选，填写数据用途。 |

多个步骤可以引用同一个数据标识。一个步骤需要多个数据时，可以用英文逗号分隔，例如 username,phone。

## 常见示例

新增用户用例：

| 表 | 推荐写法 |
|---|---|
| 用例清单 | TC001，新增用户，填写目标页面URL，预期结果写“页面提示保存成功，用户列表中显示新建用户”。 |
| 步骤明细 | “点击新增按钮，打开新增用户弹窗”“输入用户名称”“输入手机号”“点击保存按钮”。 |
| 测试数据 | username 对应“测试用户001”，phone 对应“13800000001”。 |

查询用户用例：

| 表 | 推荐写法 |
|---|---|
| 用例清单 | TC002，查询用户，填写目标页面URL，预期结果写“用户列表只显示符合查询条件的用户”。 |
| 步骤明细 | “输入用户名称查询条件”“点击查询按钮”。 |
| 测试数据 | searchName 对应“测试用户001”。 |

## 不推荐写法

| 不推荐写法 | 原因 |
|---|---|
| 把所有操作写在用例清单的备注中。 | 平台无法稳定识别步骤顺序。 |
| 只写“进入用户管理”。 | 缺少目标页面 URL。 |
| 写“点击新增”但不说明会打开弹窗。 | 后续操作所在位置不清晰。 |
| 数据引用写 username，但测试数据里没有 username。 | 步骤无法找到对应数据。 |
| 预期结果写成内部检查方式。 | 预期结果应表达业务现象。 |
