# 用例生命周期与基础检查设计

## 背景

测试人员会在业务项目开发阶段先用自然语言文档整理测试用例，后续再把大量用例导入测试平台。即使暂不接入 AI 和导入功能，平台也需要先具备用例生命周期能力，避免半成品或未审查用例直接进入运行中心。

本设计只处理用例状态、基础检查、列表筛选、运行中心过滤和必要的前端状态管理。不引入 AI，不实现自然语言导入。

## 目标

- 新增用例状态：草稿、待启用、启用。
- 所有新建和复制用例默认进入草稿。
- 运行中心只展示启用且基础检查通过的用例。
- 将现有定位检查升级为基础检查的一部分。
- 基础检查自动执行，不增加手动检查按钮。
- 项目详情页支持状态筛选、单条状态切换和批量状态切换。
- 用例编辑页在步骤表中展示基础检查问题和建议。
- 启用 Pinia，但只管理跨页面、可持久化或共享的 UI 状态。
- 实施完成后同步维护 README 和 docs 文档。

## 非目标

- 不做自然语言用例导入。
- 不接入大模型。
- 不新增专门的审查工作台页面。
- 不把页面局部 UI 状态整体迁移到 Pinia。
- 不要求启用用例必须执行过实测检查。

## 数据模型

`CaseMeta` 增加用例状态字段：

```ts
status: 'draft' | 'ready' | 'active'
```

显示文案：

- `draft`：草稿
- `ready`：待启用
- `active`：启用

历史用例缺少 `status` 时按 `draft` 兼容。

当前 `review` 字段继续保留，但语义从“定位检查”升级为“基础检查”。基础检查继续使用 `CaseReview` 和 `CaseReviewItem`，避免另起一套审查结构。后续实现可以为 `CaseReviewItem` 增加分组字段，例如 `group: 'integrity' | 'locator' | 'assertion' | 'timeout'`，用于前端展示问题来源。

复制用例时：

- 复制步骤和基础检查结果。
- 新用例状态设为草稿。
- 不复制实测检查结果。

## 用例状态规则

用例状态只通过用户明确动作改变。自动基础检查只更新检查结果，不自动把草稿提升为待启用，也不自动启用用例。

状态切换规则：

- 切到草稿：永远允许。
- 切到待启用：必须基础检查通过。
- 切到启用：必须基础检查通过。
- 实测失败不阻止启用。
- 基础检查不通过时，不能切到待启用或启用。

保存用例内容后：

- 草稿仍保持草稿。
- 待启用或启用不自动降级。
- 基础检查结果重新计算。
- 如果启用用例被改到基础检查不通过，运行中心必须排除该用例，后端运行前也必须拒绝该用例。

## 基础检查

基础检查是自动能力，不提供单独按钮。

触发时机：

- 保存并生成测试文件。
- 停止录制并写入步骤。
- 后端读取历史用例且缺少检查结果时，可以补齐基础检查结果。
- 单条或批量切换到待启用、启用前，后端必须基于最新数据校验基础检查是否通过。

不触发基础检查的场景：

- 新建空用例。新建后显示草稿和未审查。
- 复制用例。复制时直接复用基础检查结果。

基础检查包含完整性检查和定位质量检查。

完整性检查范围：

- 用例名称不能为空。
- 起始路径必须有效。
- 步骤不能为空。
- 依赖元素定位的步骤必须有 selector。
- 依赖输入值或断言值的步骤必须有 value。
- timeout 必须是整数，范围为 `0` 到 `600000` 毫秒。
- 多页面别名、弹窗打开步骤等结构必须自洽。

定位质量检查范围：

- 沿用现有动态 UUID id 检查。
- 沿用现有宽泛框架选择器检查。
- 沿用现有瞬态状态 class 检查。
- 沿用现有结构顺序选择器检查。
- 沿用现有弱角色定位检查。

基础检查通过定义：

- 不存在 `error` 或 `danger` 级别问题。
- `warning` 和 `info` 只提示，不阻断待启用或启用。

## 检查状态

项目列表展示一个合成检查状态，不把所有底层状态直接摊开。

主状态：

- `未审查`：仅用于新建空用例、历史兼容或异常兜底。
- `审查不通过`：基础检查有阻断问题。
- `待实测`：基础检查通过，但没有有效实测结果。
- `实测失败`：基础检查通过，但最近一次实测失败。
- `实测通过`：最近一次实测通过。

如果用例内容变更导致实测结果失效，列表显示为 `待实测`，并用提示说明已有实测结果已过期。

## 项目详情页

项目详情页作为用例批量管理入口。

列表调整：

- 新增 `用例状态` 列。
- 当前 `实测检查` 列改为 `检查状态`。
- `用例状态` 列支持单条下拉切换：草稿、待启用、启用。
- 状态切换失败时，前端展示后端返回的具体原因。

筛选：

- 全部
- 草稿
- 待启用
- 启用

批量操作：

- 批量设为草稿。
- 批量设为待启用。
- 批量设为启用。

批量切换规则：

- 批量设为草稿时全部允许。
- 批量设为待启用或启用时逐条校验基础检查。
- 后端返回成功数量、失败数量和每条失败原因。
- 前端展示批量处理汇总，不静默跳过失败项。

回收站不参与状态筛选，也不进入运行中心。

## 用例编辑页

当前步骤表中的“定位检查”升级为“基础检查”。

每个步骤展示该步骤相关的问题标签：

- 错误
- 高危
- 警告
- 提示

鼠标悬停标签时展示：

- 问题原因。
- 修复建议。
- 问题来源，例如完整性、定位、断言、等待时间。

页面顶部或标题区域展示当前用例状态和检查状态。用例状态可以用下拉切换，规则与项目详情页一致。

保存并生成测试文件后，后端返回最新用例数据和基础检查结果，前端立即刷新步骤检查标签。

停止录制后，后端写入步骤并生成最新基础检查结果，前端刷新用例。

实测检查前不增加基础检查按钮。后端使用最新保存后的基础检查结果和用例数据执行实测。若后端因用例状态或基础检查不通过拒绝操作，必须返回前端可展示的错误信息。

## 运行中心

运行中心只展示可运行用例：

- `status === 'active'`
- 基础检查通过

草稿和待启用不展示。启用但基础检查不通过的用例也不展示。

后端 `runProject` 必须做同样校验，不能只依赖前端过滤。若请求指定了不可运行用例，后端返回明确错误，说明用例未启用或基础检查不通过。

## Pinia 使用边界

项目已安装 Pinia，本次启用 Pinia 并在 `main.ts` 挂载。

适合放入 Pinia 的状态：

- 项目当前环境选择。
- 项目页用例状态筛选。
- 运行中心选中的用例 key。

不放入 Pinia 的状态：

- 用例编辑页当前选中步骤。
- 步骤高亮状态。
- 步骤批量编辑模式。
- 弹窗和抽屉开关。
- loading 状态。
- 表单内容。
- 录制 sessionId。

实现建议新增轻量 `project-ui` store，替代当前 `project-env.ts` 直接读写 localStorage 的方式。store 只承接跨页面、可持久化或共享的 UI 状态，不做页面局部状态集中化。

## 接口与后端校验

后端需要支持：

- 创建用例时默认写入草稿状态。
- 复制用例时默认草稿，复制基础检查结果，不复制实测结果。
- 保存用例时重新生成基础检查结果。
- 停止录制时重新生成基础检查结果。
- 单条状态切换接口，或在更新用例接口中校验状态切换。
- 批量状态切换接口。
- 运行前校验用例是否启用且基础检查通过。

后端阻断必须返回可读错误：

- 单条状态切换失败时返回阻断原因。
- 批量状态切换失败时返回每条失败原因。
- 运行中心请求不可运行用例时返回明确原因。

错误信息至少包含：

- 用例名称或 key。
- 阻断类型。
- 具体问题。
- 修复建议。
